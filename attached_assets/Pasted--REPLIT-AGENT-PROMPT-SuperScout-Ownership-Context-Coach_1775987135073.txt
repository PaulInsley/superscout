# REPLIT AGENT PROMPT — SuperScout: Ownership Context Coaching Line on Captain Cards

## CONTEXT

SuperScout’s Captain Picker cards show expected points, confidence labels, ownership percentage, fixture ticker, upside, risk, and a vibe-voiced case. We’re adding one more element: a short coaching line that tells the user what the ownership percentage means for their rank if they captain this player.

This is NOT a calculated metric displayed to the user. It’s a vibe-voiced coaching observation generated by Claude. The AI uses the differential maths internally to calibrate the tone, but the user only sees natural language.

Design principle: SuperScout is a coach, not a spreadsheet.

## TASK

### 1. Add ownership context instruction to the captain picker system prompt

Find the captain picker system prompt (the one that tells Claude how to generate captain recommendations). Add the following instruction block AFTER the existing vibe instructions:

```
OWNERSHIP CONTEXT RULE:
For each captain option, include an "ownership_context" field in your
response. This is a single short sentence (max 20 words) that tells
the user what the ownership percentage means for their rank if they
captain this player.

Guidelines by ownership band:

- 50%+: Emphasise that NOT captaining this player is the risk.
  Most of the field benefits if he hauls. Captaining him protects
  rank; skipping him is the gamble.

- 30–49%: This is a popular pick. A haul helps you, but it helps
  a lot of other managers too. This protects rank, it doesn't
  gain ground.

- 10–29%: Differential territory begins. A haul here gains ground
  on most managers who don't own him.

- 3–9%: Strong differential. Very few managers benefit if he
  delivers. A haul is a significant rank swing.

- Below 3%: Extreme differential. Almost nobody benefits but you.
  But acknowledge the expected points trade-off if this player's
  EXP PTS is meaningfully lower than the top option.

CRITICAL RULES:

1. NEVER imply a low-ownership pick is automatically better. If a
   differential pick has 3+ fewer expected points than the top
   option, the context line MUST acknowledge the trade-off. A
   differential only matters if the player delivers.

2. When two players have similar expected points (within 2 points)
   but very different ownership, highlight that the lower-owned
   player offers genuine differential upside without much sacrifice.
   That is the sweet spot.

3. If Triple Captain chip is active, the captain gets 3× points.
   Adjust context accordingly — the differential upside is amplified
   because you gain an extra multiplier the field doesn't get.

4. If expected points are below 4, the context line should gently
   discourage captaining this player regardless of ownership. Low
   floor picks are not bold — they are reckless.

5. If ownership data is missing or unavailable, omit the
   ownership_context field entirely. Do not guess.

Write the line in the active vibe voice. The Expert is measured and
analytical. The Critic is blunt and challenging. The Fanboy is
excited but honest.
```

### 2. Update the captain response schema

Add `ownership_context` to the expected response object from Claude for each captain option. It sits alongside the existing fields:

- `ownership_context` — string or null. A single vibe-voiced coaching sentence. Null if ownership data is unavailable.

Update the response parsing code to extract this field from Claude’s response. If the field is missing or null, treat it as null (no error, no fallback text).

### 3. Display on captain cards

Add the ownership_context text to each captain card in the UI:

- **Position:** Directly below the existing ownership percentage line (e.g. below “45.5% owned”)
- **Style:** Muted secondary text (text-secondary colour from the theme). Italic. Same typographic treatment as the risk line. No colour coding — no green for “good differential”, no red for “bad differential”. The text carries the meaning.
- **If null:** Render nothing. No placeholder, no empty space, no “loading” state.

### 4. Remove the old captain impact calculator

- Remove any front-end code that calculates differential impact using ownership percentage

These are being replaced by the AI-generated ownership_context line, which is smarter, personalised, and vibe-voiced.
