## FIX 1: player_id is NULL on recommendation_options — CRITICAL

**The problem:** Every row in the `recommendation_options` table has `player_id = NULL`. The AI generates recommendations that mention player names in the text (upside_text, risk_text), but the actual FPL player ID number is never stored.

**Why this matters:** Without player_id, we cannot link a recommendation to the player’s actual gameweek score later. The entire “What If” analysis, Manager Profile, and season-end learning system depend on knowing WHICH PLAYER was recommended, not just what text the AI wrote about them.

**What needs to happen:**

1. When the AI generates a recommendation option that mentions a specific player (which is every captain pick and every transfer suggestion), the system must identify that player’s FPL ID.
1. The FPL API `/bootstrap-static/` endpoint returns an `elements` array where each player has an `id` field and a `second_name` field (surname).
1. After the AI generates its response, match each recommended player name against the `elements` array using the `second_name` field. Store the matched player’s `id` as `player_id` in the `recommendation_options` row.
1. For captain picks: each option IS a player — match the player name to get their FPL ID.
1. For transfer suggestions: each option involves an “out” player and an “in” player. Store the “in” player’s FPL ID as the player_id (this is the player being recommended). If the option is “Hold Your Transfer” (no player), player_id should remain NULL — that’s correct for hold options.
1. For restructure packages (multi-transfer options): store the primary “in” player’s FPL ID, or if that’s ambiguous, store the first “in” player’s ID. This is a best-effort match — having one player_id per restructure package is better than having none.

**Matching approach:** The player data from `/bootstrap-static/` should already be cached by the API caching layer. Use the cached data rather than making a fresh API call. Match on `second_name` (case-insensitive). If there are multiple players with the same surname (e.g. two players called “Williams”), also check `first_name` or `team` to disambiguate.

## FIX 2: confidence_levels is NULL on recommendations — HIGH

**The problem:** The `confidence_levels` field on the `recommendations` table is NULL on every row. The confidence data exists on the child table (`recommendation_options` has `confidence_score` values like 0.3, 0.6, 0.9) but it’s not being written to the parent table.

**What needs to happen:**

1. When writing a new row to the `recommendations` table, also populate the `confidence_levels` JSONB field.
1. The format should be an array of objects, one per option. Example:

```json
[
  {"player": "Fernandes", "confidence": 0.9, "label": "BANKER"},
  {"player": "Palmer", "confidence": 0.6, "label": "CALCULATED_RISK"},
  {"player": "Cunha", "confidence": 0.3, "label": "BOLD_PUNT"}
]
```

1. This data is available at the point of recommendation generation — the same data being written to recommendation_options should also be summarised here.

## FIX 3: data_sources_used is NULL on recommendations — HIGH

**The problem:** The `data_sources_used` field (text array) on the `recommendations` table is NULL on every row.

**What needs to happen:**

1. When writing a new row to the `recommendations` table, populate the `data_sources_used` field.
1. For now (Intelligence Engine Level 1), every recommendation uses two data sources. Set the value to: `["fpl_api", "ai_knowledge"]`
1. This is a text array field, not JSONB. Store it as a Postgres text array.
1. Later, when Level 2 (Understat xG) is added, we will add `"understat_xg"` to this array. For now, these two values are correct for every recommendation.
