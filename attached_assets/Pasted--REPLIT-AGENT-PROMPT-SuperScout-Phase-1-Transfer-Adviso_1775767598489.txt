## BUILD 1 — FPL API CACHING LAYER

Build this FIRST because the Transfer Advisor depends on it. The FPL API has no SLA, goes down during peak periods (deadline day, live match scoring), and returns 429 rate limit errors without a retry-after header. SuperScout must never show a blank screen or crash because the FPL API is slow or down.

### Three-Tier Cache Architecture

**Tier 1: Static Data (refreshes every 6 hours)**

Cache the following FPL API data in Supabase (or a server-side in-memory cache):

- Player master data from `/bootstrap-static/` — names, teams, positions, prices, ownership %, form, total points, status flags, chance_of_playing
- Team data from `/bootstrap-static/` — club names, short names, IDs
- Fixture data from `/fixtures/` — upcoming matches, fixture difficulty ratings
- Gameweek data from `/bootstrap-static/` — current gameweek, deadlines, is_current, is_next

Refresh this data with a background job every 6 hours — NOT on every user request. When a user opens the app, serve the cached version. If the cache is empty (first run), fetch live and populate it.

**Tier 2: Semi-Live Data (refreshes every 30 min, every 2 min during live matches)**

- Live gameweek scores from `/event/{event_id}/live/` — only during active matches
- Price change data — transfer in/out counts from bootstrap-static

During live matches (detect by checking if any fixture in the current gameweek has `started: true` and `finished: false`), increase refresh to every 2 minutes. Outside match windows, every 30 minutes is fine.

**Tier 3: User-Specific Data (on login + on demand)**

- User’s squad from `/entry/{manager_id}/event/{event_id}/picks/`
- User’s transfer history from `/entry/{manager_id}/transfers/`
- User’s overall history from `/entry/{manager_id}/history/`
- User’s chips used (derived from history)
- User’s free transfers remaining (derived from transfer history)

Cache this per-user. Refresh on login and when the user explicitly opens the Transfer Advisor or Captain Picker.

### Fallback Behaviour

When the FPL API returns errors (429, 500, timeout, network error):

1. Serve cached data with a subtle “Last updated X minutes ago” indicator in the app header
1. Disable any feature that strictly requires live data (e.g., real-time price change alerts)
1. NEVER show a blank screen or crash — always show something useful from cache
1. Log the outage internally: timestamp, endpoint, error code, duration
1. If cached data is older than 24 hours, show a visible but non-blocking banner: “FPL data may be outdated — we’re working on it”

### Rate Limiting

- Maximum 1 FPL API request per 2 seconds per endpoint
- Exponential backoff on 429 responses: wait 1s, 2s, 4s, 8s, max 60s
- Queue requests and process sequentially — no parallel firing
- Share cached responses across all users — most FPL data is public and identical for everyone

### Implementation

Update the existing FPL connector module (`services/fpl/` or `connectors/fpl/`). All existing code that calls the FPL API (Captain Picker, player list, squad screen) must now go through the cache layer, not directly to the API. The cache is the single point of access for all FPL data.

## BUILD 2 — THE TRANSFER ADVISOR

This is the highest-demand feature. A user opens this screen with their current squad, and SuperScout analyses the entire player pool to recommend 3-5 transfer options — each with a player out, a player in, the expected impact, and a vibe-voiced case for the move.

### What Data the Transfer Advisor Needs

When the screen loads, gather all of this (from the cache layer, NOT direct API calls):

**User’s current squad:**

- All 15 players with: name, team, position, price, selling price (calculate using R3.06-R3.07 from the rules engine), form, total points, ownership %, status flags, upcoming fixture and FDR

**User’s transfer situation:**

- Free transfers available (1-5, calculated from transfer history)
- Current squad value (sum of selling prices)
- Money in the bank (budget remaining)
- Chips available (which chips have been used, which remain — derived from history)

**The broader player pool:**

- All players from bootstrap-static, filtered to those the user does NOT already own
- For each potential transfer target: name, team, position, price, form, total points, ownership %, upcoming fixtures (next 3-5 gameweeks), FDR ratings, status flags

**Current gameweek context:**

- Gameweek number (for season phase awareness from the strategic context)
- Deadline timestamp
- Any blank or double gameweek information

### What the AI Receives

Build a function that assembles a context object and sends it to the AI with:

1. The user’s full squad with selling prices and upcoming fixtures
1. Their free transfer count and budget
1. Their chips remaining
1. The top 30-50 transfer targets (filtered by the backend — highest form, best upcoming fixtures, most transferred-in players this week). Do NOT send all 500+ players — that’s too much context. Pre-filter intelligently.
1. The current gameweek number and season phase
1. The user’s chosen vibe prompt (from `config/vibes/vibePrompts.ts`)
1. The FPL rules reference (from `services/rules/fpl-2025-26.md`)
1. The strategic context (from `services/rules/fpl-strategy-2025-26.md`)

The transfer-advisor-specific instruction prompt should tell the AI:

“You are generating transfer recommendations for this FPL manager. Analyse their squad, upcoming fixtures, budget, and available free transfers. Return 3 to 5 transfer options. Each option must specify: one player to sell (from their squad) and one player to buy (from the available pool), the net cost or profit of the trade, whether it uses a free transfer or costs a 4-point hit, the expected points impact over the next 3 gameweeks, a confidence level (one of: BANKER, CALCULATED_RISK, or BOLD_PUNT), one clear upside sentence, one clear risk sentence, and a one-line case written in your persona voice. Mark your top recommendation as the SuperScout Pick.

VALIDATION RULES — you MUST follow these:

- The buy player’s price must not exceed the user’s budget plus the sell player’s selling price (R1.02, R3.06)
- The transfer must not result in 4 or more players from the same club (R1.03)
- The transfer must maintain correct position counts: 2 GK, 5 DEF, 5 MID, 3 FWD (R1.04) — unless it is a like-for-like position swap
- If the user has 0 free transfers, flag that every suggestion costs a 4-point hit and explain whether the expected gain justifies it (SC2.04)
- Do not recommend transferring in a player flagged as injured (chance_of_playing = 0)
- If the user has unused chips that are approaching expiry (first-half chips before GW19), mention this in your advice
- Factor in the selling price formula: the user only keeps 50% of price rises, rounded down (R3.06). Price losses are full (R3.07). Calculate the actual sell price for each player out.

If it makes sense to HOLD the transfer (save it for next week), include that as one of the options with reasoning. ‘Do nothing’ is valid advice.

Return your response as valid JSON matching this exact structure:”

Then include this JSON template:

```json
{
  "gameweek": 31,
  "free_transfers": 1,
  "budget_remaining": 0.5,
  "recommendations": [
    {
      "player_out": "Bukayo Saka",
      "player_out_team": "Arsenal",
      "player_out_selling_price": 10.2,
      "player_in": "Mohamed Salah",
      "player_in_team": "Liverpool",
      "player_in_price": 13.0,
      "net_cost": 2.8,
      "uses_free_transfer": true,
      "hit_cost": 0,
      "expected_points_gain_3gw": 4.5,
      "confidence": "CALCULATED_RISK",
      "upside": "Salah has 3 home fixtures in the next 4 GWs, averaging 8.2 pts/home game",
      "risk": "Saka's fixtures also improve from GW33 — you may want him back",
      "case": "The vibe-voiced one-liner goes here",
      "is_superscout_pick": true
    },
    {
      "player_out": null,
      "player_in": null,
      "is_hold_recommendation": true,
      "confidence": "BANKER",
      "upside": "Your squad is well-positioned for the next 2 gameweeks",
      "risk": "You lose rollover flexibility if you hold past GW33",
      "case": "The vibe-voiced case for doing nothing"
    }
  ]
}
```

**Important:** The AI must return valid JSON only — no markdown, no preamble, no backticks. Use the same robust JSON parsing (balanced-brace extraction) built for the Captain Picker. If parsing fails, retry once. If it fails again, show: “SuperScout is thinking too hard — try again in a moment.”

### Validation Layer

Before displaying recommendations to the user, run each one through the validation module (`services/validation/fpl-validator.ts`):

1. **Budget check** — can the user afford this? Calculate: money in bank + selling price of player out >= price of player in. Use the sell price formula from R3.06-R3.07.
1. **Club limit check** — does buying this player put 4+ players from the same club in the squad? Count current squad members from the buying player’s club, minus 1 if the selling player is from the same club.
1. **Position count check** — does the transfer maintain 2 GK, 5 DEF, 5 MID, 3 FWD? A midfielder-for-midfielder swap is always valid. Cross-position swaps need checking.
1. **Player availability** — is the recommended player injured (chance_of_playing = 0)? If so, filter out. If doubtful (25/50/75), keep but add a warning.

If a recommendation fails validation, do NOT show it to the user. Log the failure for debugging. If multiple recommendations fail, the AI may need re-prompting with clearer constraints.

### What the User Sees

Display the transfer recommendations as cards, using the same reusable choice architecture component from the Captain Picker. Each card shows:

- **Player Out → Player In** (large, clear, with team abbreviations)
- **Net cost** — e.g. “Costs £2.8m” or “Saves £1.2m” or “Free (like-for-like price)”
- **Transfer type** — “Uses your free transfer” or “Costs a 4-point hit” (hit cost in red)
- **Expected 3GW impact** — e.g. “+4.5 expected points over next 3 gameweeks”
- **Confidence badge** — Banker (green), Calculated Risk (amber), Bold Punt (red/orange)
- **Upside** — one sentence
- **Risk** — one sentence
- **The case** — vibe-voiced one-liner in the highlighted style

The SuperScout Pick card is highlighted with the same badge and border treatment as the Captain Picker.

If one of the options is “Hold your transfer”, display it as a card with no player swap — just the reasoning and vibe-voiced case for patience.

At the bottom of the screen, add a small text line: **“Make your transfers in the official FPL app before the deadline”** — same pattern as the Captain Picker.

At the top of the screen, show a summary bar: **“GW31 · 1 free transfer · £0.5m in bank”** — so the user immediately sees their constraints.

### Decision Log

When the Transfer Advisor generates recommendations, write to Supabase exactly as the Captain Picker does:

- One row in `recommendations` with `decision_type` = “transfer_suggestion”
- `options_shown` contains all 3-5 options as displayed
- `persona_used` records the current vibe
- `data_sources_used` = [“fpl_api_bootstrap”, “fpl_api_fixtures”, “fpl_api_picks”, “fpl_api_transfers”, “fpl_api_history”]

The auto-pull endpoint (`/api/process-decisions/{gameweek}`) should be updated to also check the user’s actual transfers after the deadline (from `/entry/{manager_id}/transfers/`) and write to `user_decisions` — recording whether they followed SuperScout’s transfer advice, made a different transfer, or made no transfer at all.

### Pre-filtering the Player Pool

The AI cannot process 500+ players. The backend must pre-filter to the most relevant 30-50 candidates before sending to the AI. Filter logic:

1. Start with all players NOT in the user’s squad
1. Remove players flagged as unavailable (chance_of_playing = 0)
1. Remove players the user cannot afford (price > budget + highest selling price in their squad for that position)
1. Score remaining players by: form (last 5 GW average) × fixture difficulty (next 3 GWs, inverted so easy = high score) × points per million (value)
1. Take the top 50 by this composite score
1. Ensure at least 5 players per position are included (so the AI has options across GK/DEF/MID/FWD)

This filtering happens server-side before the AI call. The AI sees a curated shortlist, not a data dump.

## BUILD 3 — SPORT INTEREST TABLE (Table 19)

Add one new table to Supabase. This takes 5 minutes.

**Table: sport_interest**

- `id` — UUID, primary key, auto-generated
- `user_id` — UUID, not null, foreign key to users.id
- `sport` — text, not null (e.g. “fpl”, “nfl”, “f1”, “nba”, “cricket”, “rugby”, “baseball”)
- `interest_status` — text, not null, must be one of: “interested”, “not_interested”, “active”
- `registered_interest_at` — timestamptz, nullable (when user tapped “notify me”)
- `nudge_source` — text, nullable (which trigger prompted the interest: “end_of_season”, “mid_season”, “profile_insight”, “challenge_completion”)
- `season` — text, not null (e.g. “2026-27”)
- `created_at` — timestamptz, not null, default now()

Enable Row Level Security but do not configure policies yet (same as all other tables).

This table will be used later for cross-sport discovery nudges. For now it just needs to exist.

The 2025-26 FPL season should still have live data (GW31+). Test with real data if available. If the season has ended, use mock data mode as built for the Captain Picker.

Test scenarios:

1. **Budget-constrained transfer** — user has £0.0m in bank. All suggestions must involve selling a player worth enough to fund the buy.
