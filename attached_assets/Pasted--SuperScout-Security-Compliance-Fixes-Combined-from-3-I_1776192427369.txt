# SuperScout — Security & Compliance Fixes (Combined from 3 Independent Code Reviews)

**Context:** Three independent code reviews (BROADSPEC internal, ChatGPT static review, and a full BROADSPEC v2.3 external review) all identified the same core security issue plus several supporting findings. This prompt addresses everything that must be fixed before beta testers arrive.

**Structure:** 11 fixes, grouped into 3 tiers. Do them in order. Report back after each fix before moving to the next. Do not batch.

-----

# TIER 1 — Auth & Access Control (Fixes 1–4)

These four fixes address the same root cause: user-facing API routes trust caller-supplied user_id instead of deriving it from the authenticated session. Combined with the service-role fallback, this means unauthenticated requests get full database access and authenticated users can target another user's data.

All three reviews flagged this as the #1 priority.

-----

## Fix 1: Create auth middleware

**What to build:** A reusable middleware function in a new file `artifacts/api-server/src/lib/authMiddleware.ts` that:

1. Reads the Authorization header (Bearer token)
1. Verifies the token with Supabase Auth (calls `supabase.auth.getUser(token)`)
1. If valid: attaches the verified user ID to the request as `req.verifiedUserId` and the user-scoped Supabase client as `req.userSupabase`
1. If missing or invalid: returns 401 with `{ error: "Authentication required" }`

**Example:**

```typescript
import { Request, Response, NextFunction } from "express";
import { createClient } from "@supabase/supabase-js";

export async function requireAuth(req: Request, res: Response, next: NextFunction) {
  const authHeader = req.headers.authorization;
  if (!authHeader || !authHeader.startsWith("Bearer ")) {
    res.status(401).json({ error: "Authentication required" });
    return;
  }

  const token = authHeader.slice(7);
  const supabaseUrl = process.env.SUPABASE_URL;
  const supabaseAnonKey = process.env.SUPABASE_ANON_KEY;

  if (!supabaseUrl || !supabaseAnonKey) {
    res.status(503).json({ error: "Auth service unavailable" });
    return;
  }

  const supabase = createClient(supabaseUrl, supabaseAnonKey, {
    global: { headers: { Authorization: `Bearer ${token}` } },
  });

  const { data: { user }, error } = await supabase.auth.getUser(token);

  if (error || !user) {
    res.status(401).json({ error: "Invalid or expired token" });
    return;
  }

  (req as any).verifiedUserId = user.id;
  (req as any).userSupabase = supabase;
  next();
}
```

## Fix 2: Remove service-role fallback from user requests

**File:** `artifacts/api-server/src/lib/supabaseUser.ts`

**Current problem:** `getSupabaseForRequest` falls back to the service-role client when no auth header is present. This bypasses all RLS policies.

**Change to:**

```typescript
export function getSupabaseForRequest(req: Request): SupabaseClient | null {
  const authHeader = req.headers.authorization;
  if (authHeader && authHeader.startsWith("Bearer ")) {
    const token = authHeader.slice(7);
    return getUserSupabase(token);
  }
  // No fallback to service-role for user-facing routes
  return null;
}
```

Routes that legitimately need service-role access (admin, background jobs, pre-generation batch) should call `getSupabase()` from `supabase.ts` directly. Do NOT change those routes — only the fallback behaviour.

## Fix 6: Fix admin password timing attack

**File:** `artifacts/api-server/src/routes/admin.ts`, line 122

The `crypto` module is already imported on line 28 — do not add a duplicate import.

Find where the admin password is compared using `===` (line 122: `if (password === ADMIN_PASSWORD)`) and replace with constant-time comparison:

```typescript
// Replace line 122:  if (password === ADMIN_PASSWORD)
// With:
const passwordBuffer = Buffer.from(password);
const adminBuffer = Buffer.from(ADMIN_PASSWORD);
const isValid = passwordBuffer.length === adminBuffer.length &&
  crypto.timingSafeEqual(passwordBuffer, adminBuffer);
if (isValid) {
  // ... grant access (keep existing body)
}
```

This prevents timing attacks that could leak the admin password character by character.

## Fix 7: Add max length on captain context schema

**File:** `artifacts/api-server/src/schemas/captain.ts`

The `context` field in the captain picks schema has no maximum length. Without a limit, someone could send a massive prompt and burn API credits.

**Current:**
```typescript
context: z.string().min(1, "Context is required"),
```

**Change to:**
```typescript
context: z.string().min(1, "Context is required").max(50000),
```

50,000 characters is generous for any legitimate captain context but blocks abuse.

The transfer schema (`schemas/transfer.ts`) only has `manager_id`, `vibe`, and `skip_cache` — no free-text fields, so it's fine as-is.

## Fix 8: Rate limit admin routes

**File:** `artifacts/api-server/src/app.ts`

The admin routes are mounted BEFORE the rate limiter:

```
app.use("/api/admin", adminRouter);       // line 90 — no rate limit
app.use("/api", dataRateLimiter, router); // line 91 — rate limited
```

Add rate limiting to admin routes. Use a separate, stricter limiter:

```typescript
const adminRateLimiter = rateLimit({
  windowMs: 15 * 60 * 1000,  // 15 minutes
  max: 50,                     // 50 requests per 15 min
  standardHeaders: true,
  legacyHeaders: false,
  message: { error: "Too many admin requests" },
});

app.use("/api/admin", adminRateLimiter, adminRouter);
```

## Fix 9: Add age gate to onboarding

The app requires users to be 13+ (matching FPL's own requirement and UK AADC compliance). Currently there's no age check anywhere in onboarding.

Add an age verification step to the **SignUpScreen** (`artifacts/superscout/app/onboarding/SignUpScreen.tsx`) BEFORE account creation. Two options — pick whichever is simpler:

**Option A (checkbox):** Add a checkbox: "I confirm I am 13 years of age or older" that must be ticked before the Create Account button is enabled. Store the confirmation in the `consent_events` table (this table already exists — see `ConnectFPLScreen.tsx` for usage examples).

**Option B (date of birth):** Add a date of birth picker. If the user is under 13, show a message: "SuperScout is available for users aged 13 and over" and block account creation. Do NOT store the date of birth — just use it for the check and discard it.

Option A is simpler and sufficient for App Store compliance.
