# REPLIT AGENT PROMPT — SuperScout: Captain/Transfer Cross-Check

## CONTEXT

SuperScout’s Captain Picker and Transfer Advisor currently generate recommendations independently — they don’t know what each other recommended. This creates a serious trust problem: the app can recommend captaining a player on the Captain Picker screen while simultaneously recommending selling that same player on the Transfer Advisor screen.

For example, the Captain Picker might say “Captain Bernardo — he’s in great form” while the Transfer Advisor says “Sell Bernardo — terrible fixture this week.” A user who sees both screens will immediately lose trust in the product.

This needs fixing. The two features must be aware of each other’s recommendations.

## TASK

### 1. Generate transfer advice FIRST, then pass it to the captain picker

Change the generation order so that transfer recommendations are always generated before captain recommendations. This applies to both live generation AND the pre-generation pipeline.

**For live generation:**

- When the captain picker route is called, check if transfer advice has already been generated for this user + gameweek + vibe (either in the pre-gen cache or in a temporary session store)
- If transfer advice exists: extract the list of players recommended as “OUT” (being sold) and pass this list to the captain picker prompt as additional context
- If transfer advice does NOT exist yet: generate captain picks without transfer context (current behaviour — this is the fallback)

**For the pre-generation pipeline:**

- Change the generation order: generate transfer advice for all 3 vibes FIRST, then generate captain picks for all 3 vibes
- When generating captain picks, look up the transfer advice that was just generated for the same user + gameweek + vibe
- Extract the “OUT” players from the transfer advice and pass them to the captain prompt

### 2. Add transfer context to the captain picker prompt

Add the following block to the captain picker system prompt, after the existing ownership context rules:

```
TRANSFER CONTEXT:
The Transfer Advisor has recommended the following changes for this
gameweek (same user, same vibe):

[List of OUT players from transfer advice, or "No transfers recommended" if the hold option was the top pick]

CRITICAL RULES:
1. Do NOT recommend captaining any player who appears in the OUT
   list above. If the Transfer Advisor recommends selling a player,
   they should not be a captain option. The reasoning is simple:
   if the data says sell them, the data does not say captain them.

2. If the user's current captain is in the OUT list, acknowledge
   this explicitly in the SuperScout Pick commentary. For example:
   "The Transfer Advisor suggests moving Bernardo on this week —
   which means your captaincy needs to come from elsewhere."

3. If no transfer context is available (fallback scenario), generate
   captain picks normally without this constraint. Do not mention
   transfers.
```

### 3. Handle the reverse direction (captain context in transfer advice)

This is less critical but still useful. When generating transfer advice, if captain picks have already been generated for this user + gameweek + vibe, add a note to the transfer prompt:

```
CAPTAIN CONTEXT:
The Captain Picker has recommended the following players as captain
options: [list]

RULE: If a player appears as a strong captain option (BANKER or
CALCULATED_RISK confidence), think carefully before recommending
them as a transfer OUT. It's not forbidden — a player can be a
good captain this week but worth selling for the longer term —
but if you do recommend selling a captain option, explicitly
acknowledge the tension in the commentary.
```

Since we’re generating transfers first in the pipeline, this reverse direction mainly applies to live generation when someone opens Captain Picker before Transfer Advisor.

### 4. Implementation details

**Where to store the cross-reference:**

- For pre-generation: the transfer results are already in the pre_generated_recommendations table by the time captain generation starts. Just query that table.
- For live generation: after transfer advice is generated and returned, also store a lightweight summary (just the OUT player names) in a short-lived cache or in the pre_generated_recommendations table. When the captain picker runs, check for this summary.

**What to extract from transfer advice:**

- From each transfer recommendation, extract the player names listed as “OUT” (being sold)
- Include players from ALL transfer options (not just the SuperScout Pick) — if any option suggests selling a player, that player should not be a captain recommendation
- The “Hold Your Transfer” option has no OUT players — skip it

**Performance:**

- The cross-check is a simple database query or cache lookup — it should add less than 500ms to generation time
- If the cross-check query fails or times out, fall back to generating without transfer context (don’t block the user)

### 5. Do NOT change anything else

- Do NOT modify the fixture ticker, ownership spectrum, or card layout
- Do NOT modify the staleness check or minimum loading time
- Do NOT change the transfer advisor’s recommendation logic — only add captain context to its prompt if captain picks already exist
- Do NOT change the validation layer or hallucination check
