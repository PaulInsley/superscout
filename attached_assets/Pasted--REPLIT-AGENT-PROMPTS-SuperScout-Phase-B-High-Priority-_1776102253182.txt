## PROMPT B1 — Add Zod Validation Middleware to API Routes

### Context

SuperScout’s Express API server accepts request bodies without validation. Fields like vibe, user_id, manager_id, and gameweek are extracted directly from req.body without checking they exist, are the correct type, or contain safe values. The project already has a `@workspace/api-zod` library with schema definitions — it just isn’t wired into the routes as middleware.

### Task

1. **Create a validation middleware helper** at `artifacts/api-server/src/lib/validateRequest.ts`:

```typescript
import { Request, Response, NextFunction } from 'express';
import { ZodSchema, ZodError } from 'zod';

export function validateBody(schema: ZodSchema) {
  return (req: Request, res: Response, next: NextFunction) => {
    try {
      req.body = schema.parse(req.body);
      next();
    } catch (err) {
      if (err instanceof ZodError) {
        res.status(400).json({
          error: 'Invalid request',
          details: err.errors.map(e => ({
            field: e.path.join('.'),
            message: e.message,
          })),
        });
        return;
      }
      next(err);
    }
  };
}
```

1. **Create Zod schemas** for each route that accepts a request body. If schemas already exist in `@workspace/api-zod`, use those. If not, create them at `artifacts/api-server/src/schemas/`:

```typescript
// schemas/captain.ts
import { z } from 'zod';

export const captainPicksSchema = z.object({
  vibe: z.enum(['expert', 'critic', 'fanboy']),
  user_id: z.string().uuid(),
  skip_cache: z.boolean().optional().default(false),
  context: z.object({}).passthrough(), // Allow flexible context object
});
```

Create schemas for these routes at minimum:

- POST /api/captain-picks — vibe (enum), user_id (uuid), skip_cache (boolean optional), context (object)
- POST /api/transfer-advice — manager_id (number), vibe (enum), skip_cache (boolean optional)
- POST /api/banter/generate — user_id (uuid), league_id (string)
- POST /api/report-card/generate/:gameweek — manager_id (number), vibe (enum)
- POST /api/squad-card/generate — manager_id (number), gameweek (number), vibe (enum)
- POST /api/notifications/register — user_id (uuid), token (string)
- POST /api/notifications/send — user_id (uuid), notification_type (enum), gameweek (number)
- PUT /api/notifications/preferences — user_id (uuid), preferences (object)

1. **Apply the middleware** to each route. Example:

```typescript
router.post('/captain-picks', validateBody(captainPicksSchema), async (req, res) => {
  // req.body is now validated and typed
});
```

1. **Install zod** in the api-server package if not already present:

```
cd artifacts/api-server && npm install zod
```

## PROMPT B2 — Replace Empty Catch Blocks with Meaningful Error Handling

### Context

The codebase has approximately 15 instances of `catch {}` — empty catch blocks that silently swallow errors. This means auth failures, database errors, and AsyncStorage issues are completely invisible. When something goes wrong, there’s no way to diagnose it.

### Task

1. **Search the entire codebase** for `catch {}` and `catch {` patterns (empty catch blocks with no error handling inside).
1. **Replace every instance** with meaningful error handling. The pattern should be:

```typescript
// BEFORE (bad)
catch {}

// AFTER (good)
catch (err) {
  console.warn('[ComponentName] operation description failed:', err);
}
```

Use the component or file name as the prefix so errors can be traced to their source. For example:

- In captain.tsx: `catch (err) { console.warn('[CaptainPicker] auth lookup failed:', err); }`
- In settings.tsx: `catch (err) { console.warn('[Settings] notification prefs load failed:', err); }`
- In OnboardingFlow.tsx: `catch (err) { console.warn('[Onboarding] save completion state failed:', err); }`

1. **For critical operations** where the user should know something went wrong (e.g., saving vibe preference, connecting FPL account), also set an error state that shows a brief message to the user.
1. **Verify:** Search the entire codebase for `catch {}` after all changes. The count should be zero.

### Git Safety

Do NOT run any destructive git operations. Commit with message: “reliability: replace all empty catch blocks with meaningful error logging”

## PROMPT B4 — Add Pre-Notification Consent Screen

### Context

SuperScout requests push notification permission via the iOS/Android system dialog. However, PECR (Privacy and Electronic Communications Regulations) requires informed consent for marketing-style communications. The OS dialog is generic — it doesn’t explain what types of notifications SuperScout will send. We need an in-app explanation screen before triggering the OS dialog.

### Task

1. **Create a new component** `components/NotificationConsentScreen.tsx` that shows:
- A heading: “Stay in the game”
- An explanation: “SuperScout sends you:” followed by a list:
  - Deadline reminders (2 hours before each gameweek deadline)
  - Results updates (Monday morning after all matches)
  - Price change alerts (when your players may drop in price)
  - Streak reminders (when your streak is at risk)
- “Maximum 3 notifications per gameweek. You can control each type in Settings.”
- A primary button: “Enable notifications”
- A secondary/text link: “Not now” (dismisses without requesting permission)
- Style to match the app’s dark theme
1. **Wire it into the app flow.** Currently, the app requests push permissions in `app/_layout.tsx` after onboarding completes. Change this to:
- After onboarding completes, show the NotificationConsentScreen
- If user taps “Enable notifications”, call `requestPushPermissions()` which triggers the OS dialog
- If user taps “Not now”, skip the OS dialog entirely
- Store the user’s choice in AsyncStorage so the consent screen is only shown once
- If user tapped “Not now”, they can still enable notifications later via Settings (the toggle will trigger the OS dialog at that point)
1. **Never show the OS dialog without the in-app consent screen first.** The NotificationConsentScreen is the gate.

## PROMPT B5 — Fix Admin Panel Cookie Security

### Context

The admin panel stores the password as a Base64-encoded cookie value. Base64 is encoding, not encryption — anyone who intercepts the cookie can decode it to get the password. The admin panel has access to the Supabase service role key and can read/modify any database table, so this needs proper security.

### Task

1. **Replace the Base64 cookie** with a signed cookie using Express’s cookie-parser (already installed).
1. In `artifacts/api-server/src/app.ts`, configure cookie-parser with a secret:

```typescript
app.use(cookieParser(process.env.COOKIE_SECRET || 'superscout-dev-secret'));
```

1. In `artifacts/api-server/src/routes/admin.ts`, update the login handler:

```typescript
// BEFORE (insecure)
const COOKIE_VALUE = ADMIN_PASSWORD ? Buffer.from(ADMIN_PASSWORD).toString('base64') : '';
res.cookie(COOKIE_NAME, COOKIE_VALUE, { httpOnly: true, maxAge: ... });

// AFTER (signed)
// Generate a random session token on successful login
import crypto from 'crypto';
const sessionToken = crypto.randomBytes(32).toString('hex');
// Store it server-side (in-memory Map is fine for single-server)
activeSessions.set(sessionToken, { authenticatedAt: Date.now() });
// Set as signed httpOnly cookie
res.cookie(COOKIE_NAME, sessionToken, { 
  httpOnly: true, 
  signed: true, 
  secure: process.env.NODE_ENV === 'production',
  sameSite: 'strict',
  maxAge: 4 * 60 * 60 * 1000 // 4 hours
});
```

1. Update the auth check middleware to verify against the active sessions Map instead of comparing Base64 values.
1. Add session expiry — sessions should expire after 4 hours.
1. Add `COOKIE_SECRET` to the `.env.example` file.

### Git Safety

Do NOT run any destructive git operations. Commit with message: “security: replace Base64 admin cookie with signed session tokens”
