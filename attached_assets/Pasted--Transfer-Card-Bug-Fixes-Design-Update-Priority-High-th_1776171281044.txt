# Transfer Card — Bug Fixes & Design Update

**Priority:** High — these issues make the Transfer Advisor confusing and potentially misleading.

**Design Reference:** The design mockup HTML file (index.html) is included with this prompt. Use the “Proposed” column as the pixel-level reference for spacing, colours, layout, and component structure. Ignore the “Current” column — that’s just for comparison.

There are 6 issues to fix, in this order. Do them sequentially — some depend on earlier fixes.

-----

## Fix 1: Show BOTH players’ fixtures, not just one

**Problem:** The fixture row currently shows only one set of fixtures, and it’s unclear whether they belong to the IN player or the OUT player. The whole point of the transfer card is a comparison — the user needs to see both to evaluate the move.

**What to build:**

Replace the single fixture row with two labelled rows inside the fixture section:

```
NEXT 5 FIXTURES                    🔵 Plays twice

Guéhi · MCI  [IN]
[BHA h] [EVE a] [BRE h] [BOU a] [AVL h]
─────────────────────────────────────────
Saliba · ARS  [OUT]
[MCI a] [NEW h] [FUL a] [WHU h] [BUR a]

        Easy ████████████ Hard
```

Each row has:

- A label line: “PlayerName · TEAM” followed by a small badge — green “IN” badge or grey “OUT” badge
- The 5 fixture cells for that player’s team, colour-coded by FDR
- A thin 1px divider (#F3F4F6) between the two rows

The IN player row always comes first (top). The OUT player row is second (bottom).

**Data source:** Look up fixtures for both teams from the FPL API bootstrap-static endpoint → `teams` and `fixtures`. Each fixture has `team_h`, `team_a`, and `team_h_difficulty` / `team_a_difficulty` fields. Use the difficulty value for the team in question (if Man City are playing at home, use `team_h_difficulty`; if away, use `team_a_difficulty`).

**Styling for the player label line:**

```
font-size: 10px
font-weight: 600
color: #6B7280
display: flex, align-items: center, gap: 5px
margin-bottom: 5px

IN badge: font-size 8px, font-weight 700, uppercase, color #15803D, background #F0FDF4, padding 1px 5px, border-radius 3px
OUT badge: font-size 8px, font-weight 700, uppercase, color #9CA3AF, background #F3F4F6, padding 1px 5px, border-radius 3px
```

**DGW legend:** The “🔵 Plays twice” legend in the header should only appear when at least one fixture cell in that card has a DGW. If neither player has a DGW fixture in the next 5, hide it.

## Fix 2: Fix the FDR colour mapping

**Problem:** The fixture cell colours may not match the actual FPL FDR ratings. In the current build, fixtures appear to have incorrect difficulty colours — for example, a fixture showing as easy (green) when it should be hard (red).

**What to check:** The FDR values come from the FPL API. Each fixture has difficulty ratings from 1 (easiest) to 5 (hardest). Verify that the code maps these correctly:

```
FDR 1 → background: #00875A, text: white       (dark green — easiest)
FDR 2 → background: #A3F5C1, text: #1A1A1A     (light green)
FDR 3 → background: #E7E7E7, text: #555555     (grey — neutral)
FDR 4 → background: #FF6B6B, text: white       (red)
FDR 5 → background: #80132B, text: white       (dark red — hardest)
```

**Common bug:** Using the wrong difficulty field. If Man City are playing Brighton at home, the difficulty for Man City is `team_h_difficulty` from that fixture, NOT `team_a_difficulty`. Double check the code is reading the correct field based on whether the player’s team is home or away in each fixture.

**Also check:** Blank gameweeks. If a team has no fixture in a gameweek, show a cell with “—” text on a white/light background (#F3F4F6), muted text (#9CA3AF), not an empty space or a missing cell. There must always be exactly 5 cells per row — one per gameweek.

## Fix 3: Verify and fix the impact points calculation

**Problem:** The card shows “+18 pts vs Saliba” but it’s unclear whether this number is:
(a) The differential — expected points of IN player minus expected points of OUT player (CORRECT)
(b) Just the IN player’s raw expected points (WRONG)
(c) A number the AI hallucinated in the coaching text (WRONG)

**What to check:**

1. Search the codebase for how the impact number is calculated. Look in the transfer recommendation logic — likely in the route handler or the AI prompt parsing.
1. If the AI is generating this number as part of its text output, that’s a problem — it’s not auditable or consistent. The number should be calculated deterministically from FPL data, not generated by the AI.

**Tell me what the current code does before making changes.** Then implement whichever of the following fixes are needed.

**What the number MUST represent:**

```
impact = expected_points(IN_player, N_gameweeks) - expected_points(OUT_player, N_gameweeks)
```

For hit transfers, subtract the hit cost:

```
net_impact = impact - hit_cost
```

**If expected points are not currently calculated deterministically**, implement this formula:

```
expected_points(player, N) = sum over next N gameweeks of:
  form_score × fixture_difficulty_modifier × minutes_probability
```

Where:

- `form_score` = player’s `form` field from FPL API (points per game, recent matches)
- `fixture_difficulty_modifier` = based on FDR:
  - FDR 1: 1.20×
  - FDR 2: 1.10×
  - FDR 3: 1.00×
  - FDR 4: 0.85×
  - FDR 5: 0.70×
- `minutes_probability` = based on `chance_of_playing_next_round` from FPL API. If null/missing, assume 1.0 for players with >60 mins in last 3 GWs. If 0, exclude. If 25/50/75, use 0.25/0.50/0.75.
- For DGW: the player gets TWO entries for that gameweek (one per fixture), each with its own FDR modifier.

**Projection window:**

- Free transfers: N = 3 gameweeks
- Hit transfers: N = 5 gameweeks

**Breakeven (hit transfers only):**

```
breakeven_gw = smallest N where:
  expected_points(IN, N) - expected_points(OUT, N) >= hit_cost
```

## Fix 4: Fix the form value display

**Problem:** The card showed “Form 15.0” which is impossibly high. FPL’s `form` field is typically in the range 1.0–10.0 (points per game averaged over recent matches).

**What to check:** Where does the form value displayed on the card come from? It should be the `form` field from the FPL API `elements` array in the bootstrap-static endpoint. This is a string representing points per game. If you’re using a different field (like `total_points` or `points_per_game`), switch to `form`.

The meta line should read: “MCI · £5.1m · Form 8.5” where 8.5 is the actual `form` value from the API.

## Fix 5: Ensure coaching text matches the fixture data

**Problem:** The coaching commentary references fixtures and data that don’t match what the card displays. For example, the text says “Saliba’s fixture is a nightmare (MCI away, FDR 5)” but the fixture row showed different data.

**Root cause:** The coaching text is generated by the AI prompt, and the fixture data is pulled separately. If they use different data sources or different gameweek references, they’ll contradict each other.

**What to fix:**

The AI prompt that generates coaching text must receive the SAME fixture data that the card displays. Specifically:

- Pass both the IN player’s next 5 fixtures AND the OUT player’s next 5 fixtures into the prompt
- Include the FDR rating for each fixture
- Include the calculated expected points differential
- Include whether it’s a free transfer or hit, and the breakeven if applicable

The AI should generate coaching text that is consistent with these numbers — it can add interpretation and personality, but it must not contradict the displayed data.

**Template for the data passed to the AI prompt:**

```
Transfer: [OUT player] ([team], £[price]) → [IN player] ([team], £[price])
Cost: [Free transfer / -4 pt hit]
Impact: +[N] pts over [X] gameweeks

IN player fixtures (next 5):
GW33: vs BHA (h) FDR 2 [DGW]
GW34: vs EVE (a) FDR 2
...

OUT player fixtures (next 5):
GW33: vs MCI (a) FDR 5
GW34: vs NEW (h) FDR 2
...

IN player: Form [X], Total pts [Y], Ownership [Z]%, Minutes last 3 GWs: [A, B, C]
OUT player: Form [X], Total pts [Y], Ownership [Z]%, Minutes last 3 GWs: [A, B, C]
```

This way the AI’s commentary will always match the card because it’s working from the same data.

## Fix 6: Handle blank gameweeks and DGW display

**Problem:** When a team has a blank gameweek (no fixture), the display broke — it showed “BLANK” as text but the layout was off, and fewer than 5 cells were visible.

**Rules:**

- There must ALWAYS be exactly 5 fixture cells per row, one for each of the next 5 consecutive gameweeks
- If a team has no fixture in a gameweek: show a cell with “—” text, background #F3F4F6, text colour #9CA3AF, no FDR colour
- If a team has a DGW (two fixtures in one gameweek): show both opponents separated by “/” in the cell (e.g. “BHA/BUR”), with the DGW ping indicator (indigo dot, top-right, pulsing animation). If the two fixtures have different home/away, show the first fixture’s h/a indicator.
- The coaching text should acknowledge blank gameweeks (“Saliba misses GW34 — no fixture for Arsenal”) and DGWs (“Guéhi plays twice in GW33”)

**DGW cell display:**

```
┌──────────┐
│   33    🔵│
│ BHA/BUR  │
│    h     │
└──────────┘
```
