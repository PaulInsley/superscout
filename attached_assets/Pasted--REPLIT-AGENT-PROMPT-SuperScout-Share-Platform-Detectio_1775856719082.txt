Two small fixes to the squad card share platform tracking.

**Before you start:** Do not run any destructive git operations (git reset –hard, force pushes, orphan commits) at any point during this task. If you encounter a git problem, stop and explain it to me before doing anything.

## FIX 1: SMS shares being incorrectly mapped as “twitter”

**The problem:** A user shared a Squad Card via SMS/iMessage, but the `share_platform` field in Supabase recorded “twitter” instead of “imessage”.

**The cause:** In `app/(tabs)/card.tsx`, the `mapActivityTypeToPlatform` function checks for Twitter using `at.includes("com.atebits")`. However, the matching order matters — if the check for Twitter runs before the check for Messages, or if there’s an overlap in the string matching, SMS can be mis-classified.

**What needs to change:**

1. Review the `mapActivityTypeToPlatform` function and fix the matching logic. The iOS activity types are:
- SMS/iMessage: `com.apple.UIKit.activity.Message`
- Twitter/X: `com.atebits.Tweetie2.ShareExtension` (old) or activity types containing `twitter` or `x.com`
- WhatsApp: activity types containing `whatsapp`
- Instagram: activity types containing `instagram`
- Clipboard: `com.apple.UIKit.activity.CopyToPasteboard`
- Facebook: activity types containing `facebook`
- Mail: `com.apple.UIKit.activity.Mail`
1. The matching should be specific enough that SMS/Messages never matches the Twitter check. Check for `message` ONLY in the Apple UIKit activity type pattern, not as a general substring.
1. Add a `console.log` statement that logs the raw `activityType` string whenever a share completes — this will help debug any future mis-mappings. Example: `console.log("[SuperScout] Share activityType:", activityType);`
1. Also add “facebook” detection while you’re in this function. Map any activity containing “facebook” to “facebook”. This isn’t in the database constraint yet, so also run this SQL in Supabase to add it:

```sql
ALTER TABLE squad_cards DROP CONSTRAINT IF EXISTS squad_cards_share_platform_check;
ALTER TABLE squad_cards ADD CONSTRAINT squad_cards_share_platform_check CHECK (share_platform IN ('twitter', 'whatsapp', 'imessage', 'instagram', 'clipboard', 'facebook') OR share_platform IS NULL);
```

## FIX 2: Shared cards storing NULL instead of a value for share_platform when platform is unknown

**The problem:** When sharing via `expo-sharing` (which doesn’t return an activity type), the `share_platform` is stored as NULL. But NULL is also what unshared cards have. This means we can’t distinguish “shared but platform unknown” from “not shared at all.”

**What needs to change:**

1. When a card is shared via `expo-sharing` (no activity type available), set `share_platform` to “other” instead of NULL or “unknown”.
1. “other” is not currently in the database constraint. The SQL in Fix 1 above doesn’t include it, so update the constraint to:

```sql
ALTER TABLE squad_cards DROP CONSTRAINT IF EXISTS squad_cards_share_platform_check;
ALTER TABLE squad_cards ADD CONSTRAINT squad_cards_share_platform_check CHECK (share_platform IN ('twitter', 'whatsapp', 'imessage', 'instagram', 'clipboard', 'facebook', 'other') OR share_platform IS NULL);
```

1. The rule is now: `share_platform = NULL` means “not shared”. Any non-NULL value means “shared to that platform” (or “other” if we couldn’t detect which one).
