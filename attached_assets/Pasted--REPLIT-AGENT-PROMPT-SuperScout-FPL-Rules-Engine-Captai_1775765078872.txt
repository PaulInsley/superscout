## PART 1 — CREATE THE FPL RULES REFERENCE FILE

Create a new file at `services/rules/fpl-2025-26.md` (or `.json` if you prefer structured data — but markdown is fine as it gets injected into the AI context as text).

This file contains the complete FPL 2025/26 ruleset. The AI must reference this when generating any recommendation. Copy the following rules exactly:

### Squad Composition Rules

- R1.01: Squad size is exactly 15 players (11 starting + 4 bench)
- R1.02: Budget is £100.0 million at the start of the season
- R1.03: Maximum 3 players from any single Premier League club
- R1.04: Squad must contain exactly: 2 Goalkeepers, 5 Defenders, 5 Midfielders, 3 Forwards
- R1.05: Starting XI must be exactly 11 players in a valid formation
- R1.06: Valid formations require: exactly 1 GK, minimum 3 DEF, minimum 2 MID, minimum 1 FWD
- R1.07: Common valid formations: 3-4-3, 3-5-2, 4-3-3, 4-4-2, 4-5-1, 5-3-2, 5-4-1, 5-2-3
- R1.08: Bench must contain exactly: 1 GK, 3 outfield players (any position)
- R1.09: Bench order for outfield substitutes must be ranked 1st, 2nd, 3rd by the manager

### Transfer Rules

- R2.01: Each manager receives 1 free transfer per gameweek
- R2.02: Unused free transfers roll over to the next gameweek, up to a maximum of 5
- R2.03: Additional transfers beyond free transfers cost 4 points each (a “hit”)
- R2.04: Transfer deadline is typically 90 minutes before the first match of the gameweek
- R2.05: Once a transfer is confirmed, it cannot be reversed
- R2.06: Wildcard chip overrides R2.01–R2.03 — unlimited free transfers for one gameweek (permanent changes)
- R2.07: Free Hit chip overrides R2.01–R2.03 — unlimited free transfers for one gameweek (squad reverts next gameweek)
- R2.08: NEW 2025/26 — In GW16, all managers receive free transfers topped up to the maximum of 5, to help navigate AFCON absences

### Pricing Rules

- R3.01: Player prices change daily, typically at 2:30am BST
- R3.02: Price changes are always £0.1m increments (up or down)
- R3.03: Maximum price change is £0.3m per gameweek
- R3.04: Price changes are driven by net transfer activity (transfers in minus transfers out)
- R3.05: The exact threshold for price changes is hidden by FPL (not public)
- R3.06: SELLING PRICE RULE (critical): You keep only 50% of price rises, rounded down. For every £0.2m a player rises above your purchase price, you gain £0.1m in selling price
- R3.07: Price losses are FULL — if a player drops £0.1m below your purchase price, your selling price drops by the full £0.1m immediately
- R3.08: If a player goes directly from a red flag to no flag, their price is locked for approximately 1 week

### Captaincy Rules

- R4.01: One player must be designated captain each gameweek — their points are doubled
- R4.02: One player must be designated vice-captain — their points are doubled only if the captain scores 0 points (does not play at all)
- R4.03: Captain and vice-captain must be from the starting XI (not bench)
- R4.04: If the captain does not play at all and is auto-substituted, the vice-captain’s points are doubled instead
- R4.05: If both captain and vice-captain do not play, no points are doubled

### Chip Rules — New for 2025/26

- R5.01: There are 8 chips total — 2 complete sets of 4
- R5.02: Each set contains: 1 Wildcard, 1 Free Hit, 1 Triple Captain, 1 Bench Boost
- R5.03: First set must be used before the GW19 deadline (18:30 GMT, Tuesday 30 December 2025)
- R5.04: Chips from the first set CANNOT be carried over to the second half
- R5.05: Second set is available from GW20 onwards
- R5.06: Only one chip can be active per gameweek
- R5.07: The Assistant Manager chip from 2024/25 has been REMOVED for 2025/26
- R5.08: WILDCARD — Unlimited permanent transfers for one gameweek. No point cost. Team changes are permanent.
- R5.09: FREE HIT — Unlimited transfers for one gameweek only. Squad reverts to the pre-Free-Hit team the following gameweek.
- R5.10: TRIPLE CAPTAIN — Captain’s points are tripled (not doubled) for one gameweek.
- R5.11: BENCH BOOST — All 15 players’ points count (including bench players) for one gameweek.

### Scoring System

- Playing up to 60 minutes: 1 pt (all positions)
- Playing 60+ minutes: 2 pts (all positions)
- Goal scored: GK 10, DEF 6, MID 5, FWD 4
- Assist: 3 pts (all positions)
- Clean sheet: GK 4, DEF 4, MID 1, FWD 0
- Every 3 saves (GK only): 1 pt
- Penalty save: 5 pts (all positions)
- Defensive contributions — DEF (10 CBIT): 2 pts
- Defensive contributions — MID/FWD (12 CBIRT): 2 pts
- Penalty miss: -2 pts (all positions)
- Yellow card: -1 pt (all positions)
- Red card: -3 pts (all positions)
- Own goal: -2 pts (all positions)
- Every 2 goals conceded: GK -1, DEF -1, MID 0, FWD 0
- Bonus points (BPS top 3 per match): 1-3 pts (all positions)

### Defensive Contributions — New for 2025/26

- R7.01: Defenders earn 2 FPL points for 10+ combined clearances, blocks, interceptions, tackles (CBIT) in a single match
- R7.02: Midfielders earn 2 FPL points for 12+ combined clearances, blocks, interceptions, tackles, and ball recoveries (CBIRT) in a single match
- R7.03: Forwards earn 2 FPL points for 12+ CBIRT in a single match
- R7.04: Maximum 2 defensive contribution points per player per match (no partial — it’s 0 or 2)
- R7.05: Goalkeepers are NOT eligible for defensive contribution points

### Assist Rule Changes — New for 2025/26

- R8.01: A deflected pass off a defender now counts as an assist IF: (a) the goalscorer receives the ball inside the penalty area, AND (b) there is only one defensive touch before the ball reaches the goalscorer
- R8.02: The intended target of the pass is no longer relevant if the above conditions are met
- R8.03: No assist if: the defensive touch is a pass, the goalscorer loses and regains the ball, or there are two or more defensive touches
- R8.04: Assists are now given for goals resulting from free kicks/penalties awarded for handballs
- R8.05: These changes would have resulted in 41 additional assists last season

### Bonus Points System Changes — New for 2025/26

- R9.01: Goalkeeper saves from inside the box: 3 BPS (up from 2)
- R9.02: Goalkeeper saves from outside the box: 2 BPS (unchanged)
- R9.03: Penalty save: 8 BPS (down from 9)
- R9.04: Goal-line clearance: 9 BPS (up from 3)
- R9.05: Penalty goal scored: 12 BPS (regardless of position — previously varied)
- R9.06: Successful tackles: 2 BPS each (tackles lost no longer deduct)
- R9.07: BPS tie resolution: tie for 1st = both get 3 pts, 3rd gets 1; tie for 2nd = 1st gets 3, both 2nd get 2; tie for 3rd = 1st gets 3, 2nd gets 2, tied players get 1

### Auto-Substitution Rules

- R10.01: If a starting player does not play at all in a gameweek (0 minutes, no card), they are automatically replaced
- R10.02: GK auto-sub: backup GK replaces the starting GK (if the backup played)
- R10.03: Outfield auto-sub: the highest-priority bench player who played replaces the absent starter
- R10.04: Formation constraint: auto-sub must not leave fewer than 3 DEF or fewer than 1 FWD in the starting XI
- R10.05: If a starting DEF doesn’t play AND the user is in a 3-DEF formation, only a bench DEF can replace them
- R10.06: Auto-subs happen at the end of the gameweek, not in real time
- R10.07: A player who plays 1+ minutes (even if subbed off at minute 1) is NOT auto-substituted

### Deadlines and Gameweek Structure

- R11.01: The FPL season consists of 38 gameweeks
- R11.02: Each gameweek has a deadline, typically 90 minutes before the first match
- R11.03: Standard deadline: Saturday 11:00 GMT for 15:00 kickoffs (varies for Friday/Monday/midweek)
- R11.04: All team changes (transfers, captain, bench order) must be made before the deadline
- R11.05: Blank gameweeks occur when teams don’t have a fixture (e.g., due to FA Cup clashes)
- R11.06: Double gameweeks occur when teams play twice in one gameweek
- R11.07: Double and blank gameweeks are the most strategically significant events in the FPL calendar

## PART 2 — CREATE THE STRATEGIC CONTEXT FILE

Create a second file at `services/rules/fpl-strategy-2025-26.md`. This contains strategic knowledge — not rules, but expert principles that any serious FPL manager would know.

### Chip Strategy (2025/26 Specific)

- SC1.01: BENCH BOOST is most valuable during double gameweeks when bench players have 2 fixtures
- SC1.02: TRIPLE CAPTAIN is most valuable when a premium asset (£12m+) has a favourable single fixture against a weak defence
- SC1.03: FREE HIT is most valuable during blank gameweeks when many teams don’t play, or in weeks with unusual fixture congestion
- SC1.04: WILDCARD is best used during fixture swings — when multiple teams’ difficulty changes simultaneously
- SC1.05: Having 2 sets of chips means managers should be more aggressive with first-half chips. First-half chips expire at GW19 — an unused chip is a wasted chip.
- SC1.06: The second Wildcard in the second half means managers can be more aggressive with their first-half team, knowing a full reset is available
- SC1.07: Chip combinations: Use Free Hit in a blank GW, then Wildcard the following week for maximum squad restructuring
- SC1.08: The AFCON GW16 free transfer top-up (R2.08) means managers should burn all free transfers before GW16 to maximise the benefit

### Transfer Strategy

- SC2.01: Early transfers capture price rises but risk injury before the deadline
- SC2.02: Late transfers have maximum information (team news, press conferences) but risk price drops
- SC2.03: Optimal strategy: make confirmed transfers early for price rises, hold uncertain transfers until Friday press conferences
- SC2.04: “Taking a hit” (extra -4 per transfer) is justified when the expected points gain over the next 4-6 gameweeks exceeds 4 points
- SC2.05: Rage transfers (impulsive transfers immediately after poor results) are almost always suboptimal
- SC2.06: Selling a player about to drop in price is more urgent than buying one about to rise — losses are full, profits are halved (R3.06-R3.07)

### Captain Strategy

- SC3.01: The expected points ceiling matters more than the floor for captaincy — you want the highest upside, not the safest option
- SC3.02: Ownership percentage affects captaincy in mini-leagues: captaining a highly-owned player is “safe” (you don’t lose ground), captaining a differential is “aggressive” (you gain or lose ground)
- SC3.03: Home advantage is statistically significant — home players outscore away players on average
- SC3.04: Double gameweek captains are high-value because they get two chances to score
- SC3.05: Rotation risk must be assessed for midweek fixtures — managers with Champions League commitments often rotate

### Defensive Contribution Strategy — New for 2025/26

- SC4.01: Centre-backs accumulate CBIT faster than full-backs (more clearances, blocks, interceptions)
- SC4.02: CDMs (e.g., Caicedo, Rice) accumulate CBIRT fastest among midfielders due to high tackle and recovery counts
- SC4.03: Defensive contribution points are binary (0 or 2) — near-misses give nothing
- SC4.04: Teams that defend deep and face more shots generate more CBIT opportunities — budget defenders from lower-table teams gain value
- SC4.05: Track CBIT/CBIRT rates per player to predict defensive contribution probability

### Fixture Difficulty and Scheduling

- SC5.01: FPL API provides Fixture Difficulty Rating (FDR) from 1 (easy) to 5 (hard) for each fixture
- SC5.02: FDR is a useful starting point but doesn’t update in-season based on form
- SC5.03: A player’s fixture over the next 3-5 gameweeks matters more than this week alone for transfer decisions
- SC5.04: Fixture swings (when a team goes from hard to easy or vice versa) are the primary transfer trigger
- SC5.05: Blank and double gameweeks should trigger chip planning advice

### Season Phases

- SC6.01: GW1-GW4 (Early season): Limited data. Template teams dominate. Patience is key. Avoid hits.
- SC6.02: GW5-GW10 (Form emerges): Enough data to identify genuine performers vs flukes. First Wildcard window.
- SC6.03: GW11-GW18 (First-half chips): Use remaining first-half chips. Plan around blanks/doubles. AFCON planning.
- SC6.04: GW19 (First-half chip deadline): Any unused first-half chips expire. Use or lose.
- SC6.05: GW20-GW30 (Second half): Second chip set available. New Wildcard. Transfer accumulation matters.
- SC6.06: GW31-GW38 (Run-in): Mini-league decisive. Differentials matter more. Second-half chips deployed.

## PART 3 — WIRE THE RULES INTO THE AI SERVICE

Update the AI service (the module that assembles the prompt and calls the AI) so that the FPL rules and strategic context are injected into every AI call. Here’s how:

1. **Read both files** (`services/rules/fpl-2025-26.md` and `services/rules/fpl-strategy-2025-26.md`) at server startup and cache them in memory
1. **Add the rules to the system prompt.** The AI’s system prompt should now be assembled as: `SHARED_SYSTEM_PROMPT` + `VIBE_PROMPT` + `FPL_RULES` + `FPL_STRATEGY` + `FEATURE_SPECIFIC_INSTRUCTIONS` (e.g., the captain-picker instructions)
1. **Add an instruction line** at the start of the rules section: “You have access to the complete FPL 2025/26 ruleset below. You MUST validate all recommendations against these rules. Never recommend a transfer that violates squad composition rules, a chip that has already been used, or a captain that is not in the starting XI. If a recommendation would violate a rule, do not include it — suggest a valid alternative instead.”
1. **Add the current season phase** based on the gameweek number. Look up the current gameweek from the FPL API and include the matching season phase context (SC6.01-SC6.06) prominently. For example: “Current season phase: GW31-GW38 (Run-in). Mini-league decisive. Differentials matter more.”

The sport-agnostic design principle: the shared system prompt and vibe prompt stay the same regardless of sport. The rules files are the ONLY part that changes. When F1 Fantasy is added, we create `services/rules/f1-2026.md` and swap it in. The AI service should be structured so the rules file path is configurable, not hardcoded to FPL.

## PART 4 — BUILD THE VALIDATION LAYER

Create a new module at `services/validation/fpl-validator.ts` that checks recommendations before they are shown to the user.

For now, implement these validation checks for captain recommendations:

1. **Player must be in the starting XI** (R4.03) — if the AI recommends a bench player as captain, filter them out
1. **Player must have a fixture this gameweek** — if the AI recommends a player with no fixture (blank gameweek), filter them out
1. **Player availability check** — if the FPL API flags a player as injured (chance_of_playing = 0), filter them out. If flagged as doubtful (chance_of_playing = 25 or 50 or 75), keep them but add a warning note

For transfer recommendations (which we’ll build next), the validation layer will need:

1. **Budget check** (R1.02, R3.06) — can the user afford this transfer, accounting for the sell price formula?
1. **Club limit check** (R1.03) — does this transfer put 4 or more players from the same club in the squad?
1. **Position count check** (R1.04) — does this transfer maintain the correct position distribution?

Build the transfer validation checks now even though the Transfer Advisor isn’t built yet — they should be ready to plug in.

The validation function should take a recommendation object and return either: the recommendation unchanged (if valid), the recommendation with a warning appended (if doubtful), or null (if invalid). Invalid recommendations are never shown to the user.

## PART 5 — CAPTAIN PICKER FIXES

While you’re in the codebase, fix these four issues with the Captain Picker:

### Fix A — Vibe not changing the captain advice

When the user switches their vibe in Settings (e.g., from Expert to Fanboy), the Captain Picker still generates recommendations in Expert voice.

Check and fix:

1. Settings must save the new vibe to Supabase (the `default_persona` field in the `users` table)
1. Captain Picker must read the user’s CURRENT vibe from Supabase when building the AI prompt — not from a cached or hardcoded value
1. The system prompt must actually change the AI output. Strengthen the instruction by adding: “You MUST write the case text in your assigned persona voice. The Expert is calm and analytical. The Critic is sharp and sarcastic. The Fanboy uses CAPITALS, slang like BRO and DUDE, emojis, and extreme hype. If the case text could have been written by any of the three personas, you have failed the task.”

### Fix B — Replace confidence labels

Replace HIGH/MEDIUM/SPECULATIVE throughout the Captain Picker with:

- **Banker** — the safe, obvious pick. Colour: green. Replaces HIGH.
- **Calculated Risk** — good data supports it but not guaranteed. Colour: amber. Replaces MEDIUM.
- **Bold Punt** — high ceiling, real chance it blanks. Colour: red/orange. Replaces SPECULATIVE.

Update: the AI instruction prompt (use BANKER, CALCULATED_RISK, BOLD_PUNT), the JSON template, the card display, and the Decision Log fields.

### Fix C — Remove the Confirm Captain button

The Captain Picker should NOT have a confirm button or selection state. SuperScout does not push picks to the official FPL app. Remove:

- The “Confirm Captain” button
- The vibe-voiced confirmation message
- The “Selected” state on cards
- The manual write to `user_decisions` that happened on confirm

The Captain Picker now does two things only: generates and displays 3 recommendations, and silently logs them to Supabase.

Add a small text line at the bottom of the screen: **“Set your captain in the official FPL app before the deadline”** — styled subtly, not as a button.

### Fix D — Auto-pull actual captain choices after deadline

Build a background function that populates `user_decisions` automatically after each gameweek:

1. Create an API endpoint: `/api/process-decisions/{gameweek}`
1. For each user who received captain recommendations that gameweek:
- Fetch their actual picks from `/entry/{manager_id}/event/{event_id}/picks/`
- Find which player has `is_captain: true`
- Match against the recommendation row in Supabase
- Write to `user_decisions`: `recommendation_id`, `chosen_option` (the player they actually captained), `hours_before_deadline` = null
1. Edge cases: skip users whose FPL ID returns an error, still record if they captained someone NOT in the 3 recommendations (that’s valuable data — they ignored the advice)
1. Log to console: how many users processed, how many followed a SuperScout recommendation, how many went their own way
