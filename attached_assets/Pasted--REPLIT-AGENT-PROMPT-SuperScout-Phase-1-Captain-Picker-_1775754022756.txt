# REPLIT AGENT PROMPT — SuperScout Phase 1: Captain Picker with Choice Architecture + Decision Log

This is the biggest feature build so far. The Captain Picker is where the AI personality system meets real FPL data for the first time. A user opens this screen, sees their squad, and gets three captain recommendations delivered in their chosen vibe voice — with reasoning, confidence levels, and a highlighted “SuperScout Pick” they can accept with one tap.

This prompt also sets up the silent Decision Log — every recommendation shown and every choice the user makes gets written to Supabase automatically. The user never sees this logging happening. It powers features we’re building later in the season.

**Before you start:** Do not run any destructive git operations (git reset –hard, force pushes, orphan commits) at any point during this task. If you encounter a git problem, stop and explain it to me before doing anything.

-----

## PART 1 — THE CAPTAIN PICKER SCREEN

Build a new screen called “Captain Picker” accessible from the main navigation.

When the screen loads, it should:

1. Look up the current user’s FPL Manager ID from the users table in Supabase
1. Fetch their current squad from the FPL API using `/entry/{manager_id}/event/{event_id}/picks/`
1. Fetch fixture data from `/fixtures/` to get upcoming matches
1. Fetch player stats from `/bootstrap-static/` for form, points, price, ownership percentage, and team data
1. Fetch live gameweek data from `/event/{event_id}/live/` if the gameweek is in progress

All FPL API calls must go server-side (not from the client) because of CORS restrictions. Use the existing FPL connector module in `services/fpl/` or `connectors/fpl/` — do not scatter API calls elsewhere.

Once the data is loaded, send it all as context to the AI along with the user’s chosen vibe prompt. The AI should return exactly 3 captain options in a structured format.

-----

## PART 2 — WHAT THE AI RECEIVES

Build a function that assembles a context object to send to the AI. This context must include:

- The user’s full 15-player squad with each player’s: name, team, position, current form rating, total points this season, ownership percentage, price
- The upcoming fixture for each player: opponent team, home or away, fixture difficulty rating (FDR)
- The current gameweek number
- The user’s chosen vibe (expert, critic, or fanboy)
- The deadline timestamp for the current gameweek

Send this context along with the appropriate vibe system prompt (imported from `config/vibes/vibePrompts.ts` — the single source of truth established in Phase 0) plus a captain-picker-specific instruction prompt.

The captain-picker instruction prompt should tell the AI:

“You are generating captain recommendations for this FPL manager. Analyse their squad and upcoming fixtures. Return exactly 3 captain options. For each option provide: the player name, their team, the opponent and whether it is home or away, an expected points estimate (your best estimate based on form, fixtures, and historical data), a confidence level (one of: HIGH, MEDIUM, or SPECULATIVE), the player’s ownership percentage, one clear upside sentence, one clear risk sentence, and a one-line case for picking them written in your persona voice. Mark your top recommendation as the SuperScout Pick. Do not recommend goalkeepers or defenders as captain unless their attacking stats are exceptional. Return your response as valid JSON matching this exact structure:”

Then include this JSON template:

```json
{
  "gameweek": 1,
  "recommendations": [
    {
      "player_name": "Erling Haaland",
      "team": "Man City",
      "opponent": "Chelsea (H)",
      "expected_points": 7.2,
      "confidence": "HIGH",
      "ownership_pct": 68.4,
      "upside": "Hat-trick potential at home vs weak defence",
      "risk": "Low — blanked once in 8 home games",
      "case": "The persona-voiced one-liner goes here",
      "is_superscout_pick": true
    }
  ]
}
```

**Important:** The AI must return valid JSON only — no markdown, no preamble, no backticks. Parse the response as JSON. If parsing fails, retry once. If it fails again, show the user a friendly error: “SuperScout is thinking too hard — try again in a moment.”

## PART 3 — WHAT THE USER SEES

Display the 3 captain options as cards on screen. The layout for each card:

- **Player name** (large, bold) with their team badge or abbreviation
- **Opponent and venue** — e.g. “vs Chelsea (H)”
- **Expected points** — displayed prominently
- **Confidence badge** — colour-coded: HIGH = green, MEDIUM = amber, SPECULATIVE = red/orange
- **Ownership** — e.g. “68.4% owned”
- **Upside** — one sentence, displayed clearly
- **Risk** — one sentence, displayed clearly
- **The case** — the persona-voiced one-liner, styled slightly differently (italic or a different background) so it stands out as the personality element

The SuperScout Pick card should be visually distinct — a highlighted border, a “SuperScout Pick” badge, or a subtle glow. It should be the first card shown. The user should be able to accept it with one tap.

The other two cards are fully visible (not blurred or locked) for all users including free tier. The difference between free and paid tiers affects OTHER features — not the captain picker in Phase 1.

At the bottom of the screen, add a “Confirm Captain” button. When tapped, it confirms the user’s selection (whichever card they’ve tapped) and shows a brief confirmation message in the vibe voice — something like “Locked in. Let’s see how it goes.” for Expert, or “Bold choice. Don’t come crying to me on Monday.” for Critic, or “LET’S GOOOOO!! THIS IS YOUR WEEK!! 🔥” for Fanboy.

## PART 4 — THE SILENT DECISION LOG

This is invisible to the user but critical. Every time the Captain Picker generates recommendations and every time the user confirms a choice, write data to Supabase.

**When recommendations are generated**, write one row to the `recommendations` table:

- `user_id` — the current user’s ID
- `gameweek` — the current gameweek number
- `season` — the current season string (e.g. “2026-27”)
- `decision_type` — “captain_pick”
- `options_shown` — a JSON object containing all 3 options exactly as displayed to the user (player name, expected points, confidence, ownership, upside, risk, case, whether it was the SuperScout Pick)
- `confidence_levels` — a JSON object mapping each player name to its confidence level
- `data_sources_used` — for now, always set this to: `["fpl_api_bootstrap", "fpl_api_fixtures", "fpl_api_picks"]`
- `user_tier_at_generation` — the user’s subscription tier at the time this was generated
- `created_at` — timestamp

**When the user confirms their choice**, write one row to the `user_decisions` table:

- `recommendation_id` — the ID of the recommendation row just created
- `chosen_option` — the player name they selected
- `hours_before_deadline` — calculate the number of hours between now and the gameweek deadline (this is analytically important later — it tells us whether last-minute deciders perform differently from early planners)

Do not show any indication to the user that this logging is happening. No toast messages, no loading spinners for the log writes. It should be completely silent. If the log write fails, it must not block the user experience — catch the error, log it to the console, and move on.

## PART 5 — ARCHITECTURE REQUIREMENTS

This feature must be built with sport-agnostic architecture in mind. Specifically:

- The **AI prompt assembly** (building the context object, sending it, parsing the response) should be in a shared module — something like `services/ai/` — not inside the Captain Picker screen component
- The **FPL-specific data fetching** (API calls, field mappings) stays in the FPL connector module
- The **choice architecture display** (the 3-card layout with confidence badges, upside/risk, SuperScout Pick highlighting) should be a reusable component — when I build F1 Fantasy later, the same card layout will be used for driver picks
- The **Decision Log writing** should be a shared utility — it will be used by Transfer Advisor, Captain Picker, and every future decision feature

The vibe system prompts must be imported from `config/vibes/vibePrompts.ts` — the single source of truth. Do not create separate prompt files or duplicate the prompts anywhere.
