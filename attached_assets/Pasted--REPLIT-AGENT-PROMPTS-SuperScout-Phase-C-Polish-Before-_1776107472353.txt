## PROMPT C1 — Build the Vibe Quiz

### Context

The master plan (Section 12, Screen 3) specifies a 3-question personality quiz during onboarding that maps the user’s answers to a vibe (Expert, Critic, or Fanboy). Currently the app skips the quiz and goes straight to a direct vibe selection screen (Choose Your Vibe). The quiz is the “Delight” moment in the onboarding emotional arc — it makes vibe selection feel like a personality test rather than a cold pick from a list.

### Task

1. **Create a new screen** `app/onboarding/VibeQuizScreen.tsx` with three questions, each with three tappable options:

**Q1:** “Your captain blanks on 2 points. What do you want to hear?”

- (a) “Here’s why the data still supported that pick” → Expert
- (b) “I told you that was a terrible idea” → Critic
- (c) “Forget that — look at this differential for next week!” → Fanboy

**Q2:** “It’s 2 hours before the deadline. You have one free transfer. What’s your style?”

- (a) “Show me the xG data and let me decide” → Expert
- (b) “Tell me who NOT to pick — I’ll figure out the rest” → Critic
- (c) “Who’s the exciting punt this week?” → Fanboy

**Q3:** “You just had your best ever gameweek. What do you want?”

- (a) “A breakdown of which decisions drove the score” → Expert
- (b) “Yeah but here’s why you’ll probably mess it up next week” → Critic
- (c) “LET’S GOOO! Now let’s push for top 10k!” → Fanboy

1. **Quiz flow:**

- Show one question at a time (tap an answer to advance to the next)
- Progress indicator (1/3, 2/3, 3/3)
- After Q3, calculate the result: majority vibe wins. If all three answers are different vibes, default to Expert.
- Show a **vibe reveal screen** with the winning vibe name, a one-line personality preview, and the text “You can change this anytime in Settings.”
- Below the reveal, show a “Try all three” button that takes the user to the existing Compare Vibes carousel (ChooseVibeScreen in preview mode)
- A “Continue with [Vibe Name]” button to accept the quiz result and proceed

1. **Update OnboardingFlow.tsx** to insert the quiz BEFORE the existing ChooseVibeScreen. The new flow should be:

- Step 0: Welcome
- Step 1: Connect FPL
- Step 2: **Vibe Quiz (NEW)**
- Step 3: Sign Up (email/password)
- Step 4: Beginner Check
- Step 5: What We Do
- Step 6: You’re In

The quiz replaces the direct vibe selection as the primary path. The “Try all three” option on the reveal screen is the escape hatch for users who want to override the quiz result.

1. **Styling:** Match the app’s dark theme. Questions should feel playful — use slightly larger text, generous spacing. Each answer option should be a tappable card, not a radio button. The reveal screen should feel like a moment of celebration (brief animation or transition).

## PROMPT C5 — Make System Prompt Sport-Agnostic

### Context

The shared system prompt (`services/sharedSystemPrompt.ts`) directly references “FPL”, “Fantasy Premier League”, specific FPL rules, and FPL-specific terminology. For the sport-agnostic architecture to work (required for F1 Fantasy expansion in March 2027), the system prompt should be generic with sport-specific context injected dynamically.

### Task

1. **Refactor sharedSystemPrompt.ts** to be sport-neutral. Replace all hardcoded FPL references with placeholders that are filled by the sport-specific layer:
- “FPL” → use a `sportName` parameter
- “Fantasy Premier League” → use a `sportFullName` parameter
- “gameweek” → use a `roundName` parameter
- Specific rule references → injected from the rules engine, not hardcoded in the system prompt
1. **Create a function** that builds the complete system prompt by combining the generic base with sport-specific context:

```typescript
function buildSystemPrompt(sport: string = 'fpl'): string {
  const basePrompt = getGenericSystemPrompt();
  const sportContext = getSportContext(sport); // from rules engine
  return `${basePrompt}\n\n${sportContext}`;
}
```

1. **Update all routes** that use the system prompt to call `buildSystemPrompt('fpl')` instead of using the hardcoded string directly.
1. **Test:** Generate a captain pick and transfer recommendation. Verify the output quality is identical to before — the user should notice no difference.

Do NOT run any destructive git operations. Commit with message: “architecture: make system prompt sport-agnostic”

## PROMPT C8 — Add Accessibility Labels to Interactive Elements

### Context

Interactive elements (buttons, switches, cards) don’t have accessibility labels. Screen reader users (VoiceOver on iOS, TalkBack on Android) have no context for what each element does. Apple and Google both require basic accessibility for app store approval.

### Task

1. **Audit every Pressable, Switch, and tappable element** across all screens and components.
1. **Add accessibilityLabel to each one.** Examples:

- Vibe switch in Settings: `accessibilityLabel="Change your coaching vibe"`
- Notification toggle: `accessibilityLabel="Toggle deadline reminders"`
- Share button on Squad Card: `accessibilityLabel="Share your squad card"`
- Streak badge: `accessibilityLabel="Your current gameweek streak"`
- Back buttons: `accessibilityLabel="Go back"`

- Buttons: `accessibilityRole="button"`

## PROMPT C9 — Scope Supabase Service Role to Background Jobs Only

### Context

The API server creates its Supabase client with the service_role_key, which bypasses all Row Level Security. This means every API endpoint can read/write any user’s data regardless of who’s making the request. For user-facing endpoints, the server should use the requesting user’s auth token so RLS is enforced.

### Task

1. **Keep the service_role client** for background jobs only:
- Pre-generation pipeline (preGenerate.ts)
- Outcome collection (processDecisions.ts)
- Admin panel (admin.ts)
- Notification batch sending
1. **For user-facing routes**, create a per-request Supabase client using the user’s JWT:
- Extract the auth token from the request (Authorization header or from the user_id parameter + Supabase auth verification)
- Create a Supabase client with the anon key + the user’s token
- This client will have RLS enforced — users can only access their own data
1. **Create a helper** at `artifacts/api-server/src/lib/supabaseUser.ts`:

```typescript
import { createClient } from '@supabase/supabase-js';

export function getUserSupabase(accessToken: string) {
  return createClient(
    process.env.SUPABASE_URL!,
    process.env.SUPABASE_ANON_KEY!, // Use anon key, not service role
    {
      global: {
        headers: { Authorization: `Bearer ${accessToken}` },
      },
    }
  );
}
```

1. **Update these routes** to use the user-scoped client for reads/writes that should be user-specific:
- captain.ts (reading/writing recommendations, decision log)
- transfer.ts (reading/writing recommendations, decision log)
- banter.ts (reading user leagues, writing banter)
- notifications.ts (reading/writing user preferences, notification log)
- streaks.ts (reading/writing user streaks)
- reportCard.ts (reading/writing user report cards)
- squadCard.ts (reading/writing user squad cards)
- decisionLog.ts (reading user’s decision history)
1. **Add `SUPABASE_ANON_KEY`** to the api-server environment variables if not already present (it’s the same key the mobile app uses).

### Important

This is a significant change. Test thoroughly after implementing — every user-facing route should still work, but now with RLS enforced. If any RLS policy is missing or too restrictive, the route will return empty data or errors.

### Git Safety

Do NOT run any destructive git operations. Commit with message: “security: scope service_role to background jobs, enforce RLS on user routes”
